1 线程安全问题

先说结论，当多个线程，同时对共享变量写，或者一个对共享变量读、一个对共享变量写时，就会产生线程安全问题。
如下所示，创建两个线程，一个线程用于将 counter 自增 5000 次，一个线程用于将 counter 自减 5000 次，我们期望的结果是最终两个线程执行完后，counter 为 0：

static int counter = 0;

public static void main(String[] args) throws InterruptedException {
    Thread t1 = new Thread(() -> {
        for (int i = 0; i < 5000; i++) {
            counter++;
        }
    }, "t1");
    Thread t2 = new Thread(() -> {
        for (int i = 0; i < 5000; i++) {
            counter--;
        }
    }, "t2");
    t1.start();
    t2.start();
    t1.join();
    t2.join();
    log.info(counter + "");
}

但是结果并不是 0：

19:0:33.625 [main] - -1614

造成这个结果的原因是，一条 Java 语句底层其实是由多个指令构成，这里使用的 counter++、counter--，对应的 Java 字节码为：

getstatic i // 获取静态变量 i 的值
iconst_1 // 准备常量 1
iadd // 自增
putstatic i // 将修改后的值存入静态变量 i

getstatic i // 获取静态变量 i 的值
iconst_1 // 准备常量 1
isub // 自减
putstatic i // 将修改后的值存入静态变量 i

那么就导致，当两个线程分别去执行 counter++、counter-- 时，指令交错运行，代码的执行序列发生改变，而产生了错误的结果。

2 synchronized 对象锁

2.1 用法

  • synchronized 指定一个对象作为锁，并指定一个代码块作为临界区。
  • 线程进入临界区前竞争锁，未获得锁的线程阻塞，变为 BLOCKED 状态；
  • 当执行完临界区的代码后释放锁（即使抛出异常），并唤醒阻塞的线程，它们重新竞争锁。
  • synchronized 锁是非公平的，即哪个线程抢到锁就给谁。
final Object lock = new Object(); // 使用一个对象作为锁，为了避免锁被修改，最好设置为 final

synchronized (lock) { // lock 为一个对象，作为锁
    // 临界区
}

另外 synchronized 还可置于方法前，表示整个方法块都是临界区：

// 1. 实例方法
class Test {
    public synchronized void test() {}
}
// 等价于：
class Test {
    public void test() {
        synchronized (this) {}
    }
}

// 2. 静态方法
class Test {
    public synchronized static void test() {}
}
// 等价于：
class Test {
    public static void test() {
        synchronized (Test.class) {}
    }
}

2.2 原理

2.2.1 概述

synchronized 对象锁底层有多种实现，性能由好到坏依次为偏向锁、轻量级锁、重量级锁，不过这些对用户都是透明的。

  1. 一个对象 obj 被创建时为无锁状态。
  2. 当第一个线程执行 synchronized(obj) 时，obj 将变为偏向锁。
  3. 后续再有其他线程获取 obj 锁时，obj 将升级为轻量级锁。
  4. 一旦出现线程竞争锁，obj 将升级为重量级锁。
  5. 如果线程竞争重量级锁，未获得锁的线程不会立即阻塞，而是自旋一段时间。

2.2.2 对象头的 Mark Word

Java 中每个对象有一个对象头，以 32 位虚拟机为例：

如上所示，对象头包括两部分，Klass Word 是一个指向对象的类型的 class 对象的指针，Mark Word 在对象锁中会使用到：

如上所示，在对象的不同状态下，Mark Word 保存了不同的信息：

  1. biased_lock 为 0 以及状态位 01：表示对象为无锁状态，还保存了对象的 hashcode。
  2. biased_lock 为 1 以及状态位 01：表示对象为偏向锁，还保存了持有该锁的线程 ID。
  3. 状态位 00：表示对象为轻量级锁，还保存了持有该锁的线程栈帧中 Lock Record 的地址。
  4. 状态位 01：表示对象为重量级锁，还保存了对象关联的 Monitor 对象的地址。

2.2.3 重量级锁

先看重量级锁的 Mark Word：

重量级锁依赖于操作系统提供的 Monitor 对象（称为管程），它使用操作系统的互斥量（互斥信号量、互斥锁）来实现线程互斥，由于申请互斥量需要进行系统调用，那么操作系统就会从用户态切换到内核态，因此重量级锁的性能是最差的。
当申请重量级锁 obj 时，obj 对象就会关联到一个操作系统提供的 Monitor 对象，它的 Mark Word 被设置为指向 Monitor 对象的指针。
重量级锁工作流程如下所示：

  1. 当某个线程 Thread-2 获取到重量级锁 obj 时，obj 关联的 Monitor 对象的 Owner 被设置为该线程。
  2. 这时其他线程 Thread-3、Thread-4、Thread-5 来申请锁时，就会进入到 EntryList 中被阻塞。
  3. 等到线程 Thread-2 退出临界区后，Owner 被重置为 null，即释放锁，并唤醒 EntryList 中阻塞的线程来竞争锁。
  4. WaitSet：调用 wait() 方法而阻塞的线程，详见后文。

线程竞争重量级锁时还会使用自旋来优化：即线程未获得锁时，不会立刻阻塞，而是等一段时间（自旋），在这段时间内多次尝试申请锁，如果锁被释放，就有机会占用锁。在 Java 6 之后自旋锁是自适应的，如果锁对象上的上一次自旋成功了，那么这次自旋的时间就会延长，否则就会减少。

2.2.4 轻量级锁

轻量级锁的 Mark Word：

在一些情况下，线程间可能不存在对锁的竞争，比如虽然多个线程都会使用同一个锁，但是却发生在不同时刻，即线程是交替访问临界区的。这时就可以使用轻量级锁来优化，轻量级锁依赖于 CAS 原语（Compare-And-Swap），不需要系统调用，因此其性能比重量级锁更好。
轻量级锁工作流程如下所示：

  1. 每执行一次 synchronized(obj) 获得锁，就会在线程栈帧中创建一个名为锁记录（Lock Record）的数据结构：

  1. 当线程 Thread-0 执行 synchronized(obj) 时，首先将锁记录中的 Object reference 指向 obj 对象。
  2. obj 对象一开始具有无锁状态的 Mark Word，而锁记录中保存了轻量级锁的 Mark Word，现在尝试使用 CAS 原语来将二者交换，如果成功，那么现在 obj 对象就被标计为轻量级锁，并被线程 Thread-0 持有。

==>

  1. 如果 CAS 失败，说明 obj 锁已被一个线程持有，也就是说这时 obj 对象的 Mark Word 已经是轻量级锁的 Mark Word，那么此时分两种情况：
    1. obj 锁已被其他线程占有，当前线程与之竞争，轻量级锁发生竞争需要进行锁升级，详见锁升级。
    2. 当前线程已持有该锁，现在是重入锁，则线程栈桢中再添加一个锁记录，锁记录中的 Mark Word 为 null，表示这是一个重入锁。

  1. 当线程退出 synchronized 代码块时（释放锁），如果锁记录的 Mark Word 为 null，说明是重入锁，则直接移除锁记录。
  2. 否则，尝试使用 CAS 原语将 obj 锁与锁记录的 Mark Word 交换回来：
    1. 如果成功，则锁释放成功，obj 变为无锁状态。
    2. 如果失败，说明 obj 已从轻量级锁升级为重量级锁，详见锁升级。

==>

2.2.5 偏向锁

Java 15 中偏向锁已经被废弃。
偏向锁的 Mark Word：

轻量级锁中，线程执行 sychronized(obj) 时，需要执行一次 CAS 原语，来尝试交换 obj 与锁记录中的 Mark Word，即使 obj 锁已被当前线程占有，即每次重入锁时也需要进行一次 CAS 操作，偏向锁用于优化这一点。

  1. 当第一个线程执行 synchronized(obj) 时，obj 为偏向锁，跟轻量级锁一样，会进行一次 CAS，不过此时是将 obj 的 Mark Word 设置为线程的 ID ，那么下次重入锁时就不用先 CAS，通过 Mark Word 中的线程 ID 就知道锁已被当前线程占有了。obj 锁的 Mark Word 为线程 Thread-0 的 ID 时，称 obj 锁偏向于线程 Thread-0。
  2. 并且等线程释放锁后，obj 的 Mark Word 仍为该线程的 ID，不会被重置。
  3. 如果后续其他线程获得了该锁，那么偏向锁就会升级为轻量级锁，也称为撤销偏向锁。
  4. 如果发生竞争，那么升级为重量级锁。

2.2.6 锁升级（锁膨胀）

这里讲一下轻量级锁升级为重量级锁。如果在尝试获取轻量级锁的过程中，CAS 失败，这时一种情况就是有其它线程已经占有了该锁，也就是线程竞争轻量级锁，那么轻量级锁就会升级为重量级锁。

  1. 线程 Thread-0 占有锁 obj。
  2. 当线程 Thread-1 尝试获取锁时，由于发生竞争，进入锁升级过程。
  3. 申请一个 Monitor 对象，并将 obj 对象的 Mark Word 被设置该 Monitor 对象的地址。
  4. 将 Monitor 的 Owner 设置为占有锁的线程 Thread-0，线程 Thread-1 进入 Monitor 对象的 EntryList 中阻塞。

==>

2.2.7 锁粗化和锁消除

锁消除： 编译器或运行时系统在代码优化阶段检测到某些情况下不需要进行同步的代码块，并将其对应的锁操作去除的优化技术。

  • 当编译器能够确定某个对象在多线程环境中没有共享，即不会被多个线程访问，那么对该对象的锁操作就可以被消除。
  • 当编译器能够确定某个代码块中的共享资源不会被其他线程访问，那么对该代码块的锁操作就可以被消除。
int x = 0;
final Object lock = new Object();

synchronized (lock) {
    x++; // 资源没有被多线程共享，这个时候就会发生锁消除，直接将锁去掉
}

锁粗化： 将多个连续的、独立的锁操作合并为一个更大的锁操作的优化技术，这样做可以减少锁竞争的频率，从而提高程序的性能。

final Object lock = new Object();

for(int i = 0; i < 1000; i++) {
    synchronized(lock) {/*...*/}
}
// 上面这种情况就会使用锁粗化来优化
synchronized (lock) {
    for (int i = 0; i < 1000; i++) {/*...*/}
}

2.3 wait() & notify()

2.3.1 用法

wait() 和 notify() 都是定义在 Object 上的方法，要求对象 obj 作为对象锁，并被线程占有时才能调用，即必须在 synchronized(obj) 中调用：

  • obj.wait()：
    • 阻塞当前线程，并释放锁，线程变为 WAITING 状态。
    • 还可以指定一个参数：obj.wait(1000)，线程将变为 TIMED_WAITING 状态，超过 1000ms，便被自动 notify，变为 BLOCKED 状态，等锁释放后便可参与锁竞争。
    • wait() 方法可以被 interrupt() 方法打断，并抛出 InterruptedException，并将打断标记设置为 false。
  • obj.notify()：随机通知一个之前调用 obj.wait() 而阻塞的线程，变为 BLOCKED 状态，等锁释放后便可参与锁竞争。
  • obj.notifyAll()：通知所有调用了 obj.wait() 而阻塞的线程，变为 BLOCKED 状态，等锁释放后便可参与锁竞争。

之所以调用 wait()，一般是某个条件不满足了，需要等待条件满足时，调用 notify 或 notifyAll 通知它，但是 notify 是随机通知，显然不行，而 notifyAll 会通知所有的线程，不同的线程可能等待的条件不同，那么应当使用一个循环，在线程被唤醒后重新判断条件是否满足。因此 wait & notify 正确的使用方式如下：

static boolean condition = false; // 线程 t1 等待的条件

public static void main(String[] args) {
    final Object lock = new Object();

    new Thread(() -> {
        synchronized (lock) {
            while (!condition) { // wait 外应当有一个循环判断条件是否满足
                lock.wait();
            }
        }
    }, "t1").start();

    new Thread(() -> {
        synchronized (lock) {
            condition = true; 
            lock.notifyAll(); // 条件满足了，通知 t1 线程
        }
    }, "t2").start();
}

2.3.2 原理

使用 wait & notify 肯定是存在锁竞争的，所以是重量级锁：

  • 线程调用 obj.wait() 方法后，变为 WAITING 或 TIMED_WAITING 状态，并进入 obj 关联的 Monitor 的 WaitSet 中阻塞，同时将锁释放。
  • 线程调用 obj.notify() 或 obj.notifyAll() 方法后，WaitSet 中的线程将进入 EntryList 中，变为 BLOCKED 状态，等锁被释放，EntryList 中的线程便会被唤醒，并重新竞争锁。

3 LockSupport.park() & unpark()

3.1 用法

  • LockSupport.park()：
    • 阻塞当前线程，线程将变为 WAITING 状态。
    • 可以通过 interrupt() 方法打断，但是不会抛出异常，方法直接返回。
  • LockSupport.unpark(t)：唤醒调用了 park() 而阻塞的线程 t。

如果先调用 unpark()，再调用 park()，那么线程也不会阻塞。

Thread t = new Thread(() -> {
    LockSupport.park(); // 线程 t 阻塞
}, "t");
t.start();

LockSupport.unpark(t); // 唤醒线程 t

3.2 原理

每个线程都有自己的一个 Parker 对象，包含属性 _counter、_cond、_mutex。

当执行 LockSupport.park() 时：

  1. 检查 _counter 的值，如果为 0，则阻塞当前线程，线程保存到 _cond 中。
  2. 如果为 1，则继续向下执行。
  3. 将 _counter 设为 0。

当执行 LockSupport.unpark(t) 时：

  1. 将 _counter 设置为 1。
  2. 如果 _cond 不为空，即如果线程 t 调用了 park() 而阻塞，则将其唤醒，并将 _counter 设为 0。

故不管在 park() 之前还是之后调用 unpark()，unpark() 都会生效。

4 ReentrantLock

4.1 用法

与 synchronized 的区别有：

  • 调用锁申请方法而阻塞时，可被打断，不必一直等待锁。
  • 可以设置锁申请的超时时间，超时则放弃。
  • 可以设置为公平锁。
  • 支持多个条件变量，不同于 synchronized 只有 WaitSet 这一个条件变量。

也有一些共同之处，比如都是可重入锁。
使用方法：

Lock lock = new ReentrantLock();
lock.lock(); // 获取锁
try {
    // 临界区，之所以放在这里，是保证即使抛出异常，也能在 finally 中释放锁
} finally {
    lock.unlock(); // 释放锁
}

4.2 打断锁申请

使用 lockInterruptibly() 方法申请锁，在等待锁的过程中，可以调用 interrupt() 方法打断，lockInterruptibly() 方法将抛出 InterruptedException，并将打断标记设为 false。

Lock lock = new ReentrantLock();

Thread t = new Thread(() -> {
    try {
        log.info("线程 t 等待锁");
        lock.lockInterruptibly(); // 申请锁，可被打断
    } catch (InterruptedException e) {
        log.info("线程 t 被打断: " + Thread.currentThread().isInterrupted());
        return;
    }
	/* 临界区 try...finally 省略 */
}, "t");

lock.lock();
log.info("主线程获得锁");
t.start();
t.interrupt();

18:9:13.137 [t] - 线程 t 等待锁
18:9:13.138 [t] - 线程 t 被打断: false

4.3 超时放弃申请锁

使用 tryLock() 方法申请锁：

  • 无参数：如果申请失败，则放弃申请锁，方法立刻返回。
  • 有参数：如 tryLock(1, TimeUnit.SECONDS)，表示最多等待 1s，1s 内没有申请到锁，则放弃申请锁，方法返回。
Lock lock = new ReentrantLock();

Thread t = new Thread(() -> {
    log.info("线程 t 等待锁");
    if (!lock.tryLock()) {
        log.info("获取立刻失败，返回");
        return;
    }
	/* 临界区 try...finally 省略 */
}, "t");

lock.lock();
log.info("主线程获得锁");
t.start();

18:55:36.990 [t] - 线程 t 等待锁
18:55:36.990 [t] - 获取立刻失败，返回

4.4 公平锁

默认情况下，ReentrantLock 是非公平锁，即线程会去竞争锁；也可以指定为公平锁，即按照线程申请锁的顺序分配锁。

Lock lock = new ReentrantLock(true); // 只需构造函数设置参数为 true

但是公平锁一般没有必要，会降低并发度。

4.5 条件变量

在 synchronized 中，如果线程因为某个条件不满足而不能继续执行，需要调用 wait() 方法，释放锁并进入 WaitSet 中阻塞，这里的 WaitSet 就可以看作是一个条件变量，只不过，synchronized 只有一个条件变量，因此它只能要么通过 notify() 随机通知一个线程、要么通过 notifyAll() 通知所有线程，而不能指定通知等待某个条件的线程。
在 ReentrantLock 中就可以做到这一点，它支持多个条件变量，线程可以指定阻塞在某个条件变量中，也可以唤醒阻塞在指定条件变量中的线程。
使用方法如下：

static boolean hasBreakfast = false;
static Lock lock = new ReentrantLock();
static Condition waitForBreakfast = lock.newCondition(); // 条件变量，对应 hasBreakfast=true 这一条件

public static void main(String[] args) throws InterruptedException {
    Thread t = new Thread(() -> { // 线程需要等待条件 hasBreakfast==true 才能继续执行
        lock.lock();
        try {
            while (!hasBreakfast) { // 这里之所以用 while，是因为 await() 可以被 interrupt() 打断，所以唤醒后需要再检测
                try {
                    log.info("等待 breakfast");
                    waitForBreakfast.await(); // 线程阻塞在条件变量 waitForBreakfast 中，并释放锁
                } catch (InterruptedException e) {
                    log.info("await 被打断");
                }
            }
            log.info("拿到 breakfast");
        } finally {
            lock.unlock();
        }
    }, "t");
    t.start();
    
    Thread.sleep(1000);

    lock.lock(); // 线程 t 释放锁，主线程得以获得锁
    try {
        log.info("breakfast 准备好了");
        hasBreakfast = true;
        waitForBreakfast.signalAll(); // 条件 hasBreakfast==true 满足，唤醒阻塞在条件变量 waitForBreakfast 中的所有线程
    } finally {
        lock.unlock();
    }
}

20:13:52.477 [t] - 拿到 breakfast
